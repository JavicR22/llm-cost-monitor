## Lo que se construyó

### 1.1 — Monorepo inicializado
- Estructura `frontend/` + `backend/` en un solo repositorio
- `.gitignore` configurado para Python, Node, .env, IDE
- `docker-compose.yml` con PostgreSQL 15 + Redis 7 + backend, con healthchecks

### 1.2 — FastAPI con arquitectura limpia
- `app/main.py` con lifecycle hooks, CORS y routers registrados
- `app/core/config.py` con pydantic-settings (lee desde `.env`)
- Estructura de 3 capas: `api/` → `services/` → `models/`
- Todos los subdirectorios con `__init__.py`

### 1.3 — Next.js frontend scaffoldeado
- Next.js 14 + TypeScript + Tailwind CSS + App Router
- `lib/utils.ts` — función `cn()` (clsx + tailwind-merge)
- `lib/design-tokens.ts` — tokens de color del design system (navy #0F172A, blue #3B82F6, etc.)
- `lib/api.ts` — cliente HTTP tipado con manejo de errores
- `globals.css` con variables CSS del tema dark
- Directorios `components/ui/`, `components/layout/`, `components/dashboard/`, `hooks/`
- `components.json` para Shadcn UI

### 1.4 — SQLAlchemy models (15 tablas)
Modelos creados con SQLAlchemy 2.0 style (`Mapped`, `mapped_column`):

| Archivo | Modelos |
|---------|---------|
| `models/organization.py` | `Organization` |
| `models/user.py` | `User` |
| `models/api_key.py` | `ServiceAPIKey`, `ProviderAPIKey` |
| `models/provider.py` | `Provider`, `LLMModel` |
| `models/usage_log.py` | `UsageLog` (JSONB para tags, índice en created_at) |
| `models/intelligence.py` | `ModelBenchmark`, `OptimizationSuggestion`, `ShadowTestResult` |
| `models/alert.py` | `AlertRule`, `AlertEvent`, `NotificationChannel` |
| `models/audit_log.py` | `AuditLog` (sin CASCADE — inmutable) |
| `models/tag.py` | `Tag` |

Decisiones técnicas:
- `LLMModel` en vez de `Model` para evitar conflicto con SQLAlchemy
- `task_type` enum definido una sola vez con `create_type=False` en los modelos que lo reusan
- `AuditLog` sin `ondelete CASCADE` — los audit trails nunca se borran

### 1.5 — Alembic + primera migración
- `alembic/env.py` configurado para leer `DATABASE_URL` del entorno y convertir asyncpg → psycopg2 para el runner de migraciones (sync)
- Migración `initial_schema` generada con `--autogenerate`: detectó 15 tablas y 20 índices
- `alembic upgrade head` aplicado exitosamente en la DB real

### 1.6 — KeyVault (Fernet + SHA-256)
Archivo: `app/services/security/key_vault.py`

- `KeyVault.encrypt/decrypt` — Fernet para provider keys y configs de notificaciones
- `KeyVault.generate_service_key()` → `(raw_key, sha256_hash, prefix)`. La raw key nunca se persiste
- `KeyVault.hash_service_key()` — valida keys entrantes contra el hash en DB
- `get_key_vault()` — singleton `@lru_cache`, instanciado una vez al arrancar
- **11 tests unitarios** — roundtrip, key equivocada, ciphertext alterado, unicidad, formato SHA-256

### 1.7 — Auth: register, login, JWT RS256
Archivos creados:
- `app/core/database.py` — engine async + `AsyncSessionFactory` + `get_db()`
- `app/core/security.py` — bcrypt (directo, sin passlib), `create_access_token`, `create_refresh_token`, `decode_token` RS256
- `app/core/dependencies.py` — `get_current_user`, `require_role()`, `get_redis()`, alias `CurrentUser`/`DB`
- `app/schemas/auth.py` — schemas de request/response
- `app/services/auth/auth_service.py` — register (crea org + owner), login, refresh
- `app/api/v1/auth.py` — `POST /register`, `POST /login`, `POST /refresh`, `GET /me`
- `app/api/v1/router.py` — agrega todos los routers de v1

Nota: se reemplazó `passlib` por `bcrypt` directo — passlib tiene bugs de compatibilidad con bcrypt >= 4.0 en Python 3.12.

**9 tests unitarios** — hashing, verify, JWT encode/decode, token tampering

### 1.8 — Middleware
Archivos:
- `app/middleware/security_headers.py` — HSTS, X-Frame-Options, X-Content-Type-Options, CSP, Referrer-Policy, Permissions-Policy
- `app/middleware/rate_limit.py` — sliding window Redis (60s), solo en proxy endpoints, límites por plan (Free=60, Starter=200, Pro=1000, Enterprise=5000), retorna `X-RateLimit-Remaining`

Orden en `main.py` (Starlette aplica en reversa):
```
CORS → RateLimit → SecurityHeaders → handler
```

**5 tests unitarios** — headers presentes, bajo límite, sobre límite, límite exacto, plan pro

### 1.9 — Docker Compose completo
- `backend/Dockerfile` + `backend/.dockerignore`
- Stack completo verificado: postgres → redis → backend (con hot reload)
- Health check: `GET /health` → `{"status":"ok"}`
- Todos los security headers presentes en respuestas
- `POST /api/v1/auth/register` end-to-end → JWT RS256 generado con DB real

Fixes encontrados durante la integración:
- `CORS_ORIGINS` debe ser JSON array en env var: `["http://localhost:3000"]`
- `pydantic[email]` necesario para usar `EmailStr` (no incluido en pydantic base)

## Notas técnicas para recordar

- **Migraciones**: `DATABASE_URL` con driver `postgresql+asyncpg://` — alembic lo convierte a psycopg2 internamente
- **JWT keys en .env**: las claves PEM van en una sola línea con `\n` literal: `-----BEGIN RSA PRIVATE KEY-----\nMII...`
- **Docker**: el `environment:` en compose sobreescribe el `env_file:` para las mismas variables
- **Monorepo en GitHub**: un solo repo, Vercel apunta a `/frontend`, Railway a `/backend`
